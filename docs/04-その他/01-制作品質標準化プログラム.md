# 制作品質標準化プログラム — ポリッシュ・ディレクター作業標準マニュアル（2026-03-07）

# 概要

このマニュアルは、WEB制作チームの「誰がやっても一定の品質」を実現するための作業標準です。3/10週のポリッシュ充足に先立ち、初日から迷わず働ける状態を作ります。

対象者:

- **ポリッシュ**（AI制作の制作担当）
- **Wixデザイナー**（Wix制作の制作担当）
- **ディレクター**（品質チェック・顧客対応の管理者）

---

# Part 1: ポリッシュの1日の標準フロー

## 朝（9:00）担当確認

制作進行シートを開き、今日自分がやるべきことを把握します。

**確認する3つの列:**

| 列名 | 見るべきもの | アクション |
| --- | --- | --- |
| **ball_holder**（ボール保持者） | 「ポリッシュ」になっている案件 | → これが今日の自分の仕事 |
| **overall_status**（総合ステータス） | 「デモサイト制作中」「修正対応中」など | → 作業内容の把握 |
| **latest_instruction_url**（最新修正指示URL） | 最新の修正指示のリンク | → 何をすればいいかの確認 |

**優先順位の決め方:**

1. 期限切れ（delay_flag = ON）の案件 → 最優先
2. 「修正対応中」の案件 → 高優先
3. 「デモサイト制作中」「本制作中」 → 通常
4. 「新規着手待ち」 → 余力があれば

## 午前（9:30-12:00）修正対応

**修正対応の手順:**

1. **指示の確認** — `latest_instruction_url` を開いて原文を読む
2. **解釈** — 何をどう変えるかを自分の言葉で整理する
3. **不明点があれば** → すぐディレクターに確認（後回しにしない）
4. **作業実行** — 修正を行う
5. **シート更新** — `revision_memo` に何をやったかメモを残す

> **重要: 「迷ったら1人で悩まない」**
> 

> デザイン方針、修正の解釈、技術的な実装方法…迷ったらまずClaude Codeに聞く。Claude Codeで解決しない場合はディレクターにエスカレーション。
> 

## 午後（13:00-17:00）新規制作 / 修正続行

**新規制作の手順:**

1. **ヒアリングシートを読む** — `hearing_sheet_url` からアクセス
2. **業種を特定する** — 業種によって勝ちパターンが異なる
3. **Claude Codeに相談** — `/web-production [業種名]` で過去の成功パターンを参照
4. **参考サイト分析** — デザイン方針を決定
5. **制作実行** — Gemini Canvas / Cursorで制作
6. **セルフQC** — 下記の4項目チェックを実施

## 夕方（17:00-17:30）完了報告

1. 制作進行シートを更新:
    - `overall_status` → 「レビュー待ち」に変更（ディレクターチェック待ちの場合）
    - `ball_holder` → 「ディレクター」に変更
    - `revision_memo` → 作業内容のメモを記入
2. 未完了タスクがあれば、翌日の優先事項としてメモ

# Part 2: ディレクターの1日の標準フロー

## 朝（9:00）リスクチェック

制作進行シートを開き、今日注意すべき案件を把握します。

**確認する4つの列:**

| 列名 | チェック内容 | アクション |
| --- | --- | --- |
| **delay_flag**（遅延フラグ） | ONになっている案件 | → 即対応が必要 |
| **risk_level**（リスクレベル） | Critical / Highの案件 | → 原因特定と対策 |
| **overall_status** | 「ディレクターチェック中」の案件数 | → 今日のレビュー対象 |
| **ball_holder** | 「ディレクター」の案件数 | → 自分がボールを持っている案件 |

**優先対応順位:**

1. 遅延 + リスクCriticalの案件 → 即対応
2. 「ディレクターチェック中」の案件 → 品質チェック
3. 「顧客最終確認中」で長期化している案件 → フォローアップ

## 午前〜午後: レビュー & 顧客対応

**レビューの判断基準:**

| 判定 | 条件 | アクション |
| --- | --- | --- |
| **合格** | セルフQC 4項目全てOK + デザイン方針と一致 | `qc_director_pass` = ON → `overall_status` を次工程へ |
| **差戻し** | QC NG項目あり / デザイン方針と不一致 | `ball_holder` = 「ポリッシュ」に戻す + `revision_memo` に差戻し理由を記載 |

**差戻し時のルール:**

- 「だめ」だけでなく「何をどう直すか」を具体的に書く
- `revision_memo` に必ず記入（口頭だけで伝えない）
- 同じ理由で2回以上差戻す場合 → ポリッシュと一緒に画面を見て指導

**顧客からの修正依頼が来たとき:**

1. 制作進行シートの該当案件を更新:
    - `latest_instruction_url` → 修正指示のURLを貼る
    - `latest_instruction_date` → 今日の日付
    - `latest_instruction_source` → LINE / メール / 電話 など
    - `revision_count` → +1する
    - `ball_holder` → 「ポリッシュ」に変更
2. ポリッシュに作業指示を出す

## 夕方（17:00）進捗確認

1. ポリッシュ別の担当件数を確認 → 負荷の偏りがないか
2. 翌日の優先案件を確認
3. 週末: 今週の完了件数・遅延件数・手戻り率を確認

# Part 3: セルフQCチェックリスト（ポリッシュ向け 4項目）

web-productionスキルの24項目フルチェックから、ポリッシュが納品前に必ず確認する4項目です。これを通過してからディレクターに回します。

| No. | チェック項目 | 具体的な確認方法 | シート列 |
| --- | --- | --- | --- |
| 1 | **連絡先確認** | 電話番号・住所・メールアドレスがヒアリングシートと一致しているか | `check_contact` |
| 2 | **リンク確認** | 全ページのリンクを実際にクリックして正常に遷移するか | `check_links` |
| 3 | **フォーム確認** | 問い合わせフォームを実際に送信して、通知が届くか | `check_form` |
| 4 | **Google設定確認** | GA・GSCのタグが埋まっているか（該当案件のみ） | `check_google` |

**ルール:**

- 4項目全てONにしてからディレクターに回す
- 1項目でも確認できない場合はディレクターに相談
- 「たぶん大丈夫」でONにしない。必ず実際に確認する

# Part 4: Claude Code活用ガイド（ポリッシュ向け）

## いつ使うか

| 場面 | 使い方 | 具体例 |
| --- | --- | --- |
| **新規制作のデザイン方針** | `/web-production [業種名]` | 「/web-production 歯科医院」→ 過去の成功パターンが出る |
| **修正依頼の解釈** | 原文を貼って相談 | 「この修正依頼、具体的に何をすればいい？」 |
| **コーディングで詰まったとき** | エラーや要件を貼って相談 | 「このエラーを解決したい」「レスポンシブが崩れる」 |
| **デザイン選択に迷ったとき** | 選択肢を提示してもらう | 「FVのA案とB案、どちらがこの業種に合う？」 |

## 使わない場面

- 顧客とのコミュニケーション内容の判断 → ディレクターに確認
- 納品可否の最終判断 → ディレクターが行う
- セキュリティに関わる操作（APIキー・パスワード等） → 絶対にやらない

# Part 5: エスカレーションルール

| 状況 | 判断 | アクション |
| --- | --- | --- |
| 修正依頼の意味が分からない | ポリッシュ→ディレクター | Claude Codeに聞いても解釈できない場合、その旨を添えてディレクターに確認 |
| デザイン方針が決められない | ポリッシュ→ディレクター | 選択肢（A案/B案）を整理してから相談すると判断が早い |
| 技術的に実現できない | ポリッシュ→ディレクター | 「何を試したが、何ができなかった」を具体的に伝える |
| 顧客からクレームが来た | ディレクターが対応 | ポリッシュは直接顧客対応しない。ディレクターに即報告 |
| 納期に間に合わない | ポリッシュ→ディレクター | 「間に合わない」と分かった時点で即報告。納期当日に言わない |

# Part 7: ディレクター向け — 24項目フル品質チェックリスト

ポリッシュのセルフQC（4項目）を通過した後、ディレクターが確認する全項目です。

## A. デザイン品質（7項目）

- [ ]  A-1. FVが参考サイトの方向性と一致
- [ ]  A-2. カラースキームがデザイン方針と一致
- [ ]  A-3. フォントがデザイン方針の印象と合致
- [ ]  A-4. 余白・行間が適切
- [ ]  A-5. 画像がすべて適切なものに差し替え済み
- [ ]  A-6. CTAボタンが目立つ色・サイズ・配置
- [ ]  A-7. ロゴ・社名の表記が正確

## B. コンテンツ品質（7項目）

- [ ]  B-1. ヒアリング情報の主要項目がすべて反映
- [ ]  B-2. テキストに誤字脱字なし
- [ ]  B-3. ダミーテキスト残存なし
- [ ]  B-4. 電話番号・住所・メールアドレスが正確
- [ ]  B-5. 会社名・サービス名が正式名称
- [ ]  B-6. title/meta descriptionが適切
- [ ]  B-7. 汎用的すぎる見出しが具体化済み

## C. 技術品質（7項目）

- [ ]  C-1. 全ページエラーなし表示
- [ ]  C-2. 全リンク正常遷移
- [ ]  C-3. フォーム送信正常動作
- [ ]  C-4. 画像表示崩れなし
- [ ]  C-5. LCP 2.5秒以内
- [ ]  C-6. OGP設定済み
- [ ]  C-7. favicon設定済み

## D. スマホ対応（3項目）

- [ ]  D-1. スマホでレイアウト崩れなし
- [ ]  D-2. テキスト12px以上
- [ ]  D-3. ボタン44px以上

# Part 8: 制作進行シート — 役割別必須カラム早見表

## ポリッシュが毎日見る・更新する列

| グループ | 列名 | 操作 |
| --- | --- | --- |
