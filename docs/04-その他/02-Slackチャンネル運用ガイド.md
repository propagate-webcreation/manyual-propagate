# Slackチャンネル運用ガイド

## このガイドの目的

プロパゲートのマーケティングナレッジシステムは **「Slackに書くだけで、あとはAIが全部やる」** という設計思想で作られています。

ただし、それが機能するためには **「正しいチャンネルに、正しい情報を書く」** ことが前提です。

施策を `#marketing-ops-log` に書けばAIが自動でナレッジ化する。別のチャンネルに書いたら何も起きない。

このガイドは、どのチャンネルに何を書けばいいかを明確にするためのものです。

最終更新: 2026-03-20

---

# チャンネル全体マップ

```jsx
マーケティング部門
├── サブスクAd事業
│   └── #subad-ops .................. ダッシュボード改善レビュー集約
├── 集客エージェント事業
│   ├── #集客エージェント-general ... チーム全体
│   └── #集客エージェント-赤坂p .... 赤坂プロジェクト
└── ナレッジ基盤（全事業共通）
    ├── #marketing-knowledge ....... ナレッジ全般・周知
    ├── #marketing-ops-log ......... 施策記録（Bot対話）★メイン
    ├── #marketing-knowledge-input .. 外部ナレッジ投入（Bot対話）
    ├── #marketing-knowledge-qa .... Q&A（AI回答）
    └── #marketing-knowledge-review  KB昇格レビュー通知
```

---

# 共通ルール

1. **チャンネルの使い分けを守る**: 施策記録は `#marketing-ops-log`、外部ナレッジは `#marketing-knowledge-input`、質問は `#marketing-knowledge-qa`
2. **直接入力は不要**: ナレッジの蓄積はAIが全自動で行う。Slackに投稿するだけでOK
3. **レビューフローはAI起点**: KPI取得→成果判定→担当者確認→松岡レビュー依頼まで全てBotが実行
4. **Bot通知は必ず対応**: `#marketing-knowledge-review` に来たKB昇格依頼は松岡が判断

## `#marketing-ops-log` — 施策記録チャンネル

### 概要

マーケターが施策を投稿するメインチャンネル。

施策記録Bot（2分間隔で巡回）が自動検出し、**必須5項目**が揃えば即時記録、不足があればスレッドで案内。

### 必須5項目

| 項目 | スプシ列 | 説明 |
| --- | --- | --- |
| レコード番号 | C列 | kintoneのレコード番号（例: `16890`） |
| 施策カテゴリ | H列 | クリエイティブ変更 / ターゲティング変更 / 入札戦略変更 / 配信面変更 / 予算配分変更 / LP変更 |
| 施策内容 | J列 | 具体的に何をしたか（20文字以上） |
| 対象KPI | K列 | CPA / CV数 / CTR / ROAS / CVR / IMP |
| 計測期間 | L列 | 成果計測までの日数: 7 / 14 / 21 / 30日 |

### 推奨フォーマット（ピン留め済み）

```jsx
レコード番号: 16890
カテゴリ: クリエイティブ変更
KPI: CPA
計測期間: 14
内容: ASCキャンペーンのクリエイティブを3パターン追加。静止画→UGC風動画に切り替え。
```

### Botの動作フロー

**パターンA: 全5項目揃っている → 即時記録**

```
✅ 記録しました (#3)

レコード番号: 16890
媒体: Meta広告          ← 自動取得
施策カテゴリ: クリエイティブ変更
対象KPI: CPA
計測期間: 14日

📎 スプレッドシートで確認
14日後にKPI Before/Afterを自動取得し、成果を判定します
```

**パターンB: 不足項目がある → スレッドで案内**

```
📝 施策を検出しました

✅ 施策カテゴリ: クリエイティブ変更
✅ 施策内容: 検出済み

⬇️ 以下をこのスレッドに返信してください:
• レコード番号: kintoneのレコード番号（例: レコード番号: 12）
• 対象KPI: CPA / CV数 / CTR / ROAS / CVR
• 計測期間(日): 成果を計測するまでの日数（7 / 14 / 21 / 30）

全項目が揃い次第、スプレッドシートに自動記録します
```

48時間以内に全項目が揃わない場合、自動で取り消されます。

### 解像度チェック（Deep-Diveフロー）

必須5項目が揃った後、Botは **施策内容（J列）の解像度** を自動チェックします。

ナレッジとは「他の誰かが同じ状況で同じことを再現できる粒度」です。曖昧な記述はAIのレコメンデーションに使えないため、Botが具体性を確認します。

**チェック基準**: 最低30文字以上 + カテゴリ別キーワード

**フロー:**

1. 必須5項目回収完了
2. Botが施策内容の解像度をチェック
3. **十分な場合** → そのまま記録（フォローアップなし）
4. **不足の場合** → カテゴリに応じた具体的なフォローアップ質問を送信
5. ユーザーがスレッドで回答 → 施策内容を補完して記録

**カテゴリ別フォローアップ質問:**

- **入札戦略変更**: 変更前の戦略、変更後の戦略・目標値
- **クリエイティブ変更**: 変更要素（画像/動画/テキスト/CTA）、訴求軸の変化
- **ターゲティング変更**: 変更前後のオーディエンス設定
- **予算配分変更**: 変更前後の配分（金額 or 比率）
- **配信面変更**: 変更前後の配信面
- **アカウント構成変更**: 何をどう再編したか
- **LP変更**: 変更箇所 + 変更内容
- **新規施策追加**: 施策の全体像
- **その他**: 第三者が再現できるレベルの説明

**NG例 vs OK例:**

| カテゴリ | NG（解像度不足） | OK（再現可能） |
| --- | --- | --- |
| 入札戦略変更 | TCPA設定が有効 | 手動CPC ¥200 → tCPA ¥3,000に変更。学習期間2週間で安定 |
| クリエイティブ変更 | クリエイティブを改善した | 静止画3枚 → UGC風縦型動画(15秒)に差替え。訴求軸を価格→導入事例に変更 |
| ターゲティング変更 | ターゲットを変更した | 興味関心(美容全般) → LAL(CV類似1%)に切替え。除外: 既存顧客リスト追加 |
| 予算配分変更 | 予算を増やした | Meta検索 ¥50,000/日 → ¥30,000/日に減額、浮いた¥20,000をASCに配分 |

> **Botは1回だけフォローアップ質問を送ります。回答後に記録が完了します。**
> 

### 記録後の全自動フロー（KB昇格判定）

```
① 記録完了 (status: 記録中)
   ↓ 計測期間(L列)の日数が経過
② KPI取得Bot（毎日11:00）が日次実績DBからBefore/After自動取得
   → M列(Before), N列(After)に書き込み
   ↓
③ スプシ数式が自動計算
   → O列: 変化率(%), P列: 判定(○△×)
   ↓
④ 施策記録Bot（2分間隔）がP列=○を検出
   → 担当者にスレッドで確認: 「この成果判定は正しいですか？」
   → status: 担当者確認中
   ↓
⑤ 担当者がスレッドに「○」と返信
   → status: KB昇格待ち
   ↓
⑥ KB昇格レビュー通知Bot（平日10:00）
   → 松岡DMにKB昇格レビュー依頼
   ↓
⑦ 松岡がS列で「承認」を選択
   → status: KB承認済
   ↓
⑧ Notion同期Bot（30分間隔）
   → ナレッジベースDBに自動同期
```

**成果が△×の場合**: 自動で「完了」に。KBには昇格しません。

### ステータスフロー（R列）

```
記録中 → 担当者確認中 → KB昇格待ち → KB承認済
                                   ↗
成果△×の場合: 記録中 → 完了（KB昇格なし）
```

### 自動取得される項目（入力不要）

レコード番号を入力するだけで、SubAd顧客マスタから以下が自動取得されます:

- D列: 顧客名
- E列: 媒体
- F列: 業種
- G列: 担当者

その他の自動項目:

- A列: knowledge_id (PB-0001形式、自動採番)
- M列: KPI Before（日次実績DBから自動取得）
- N列: KPI After（日次実績DBから自動取得）
- O列: KPI変化率（M/N列から自動計算）
- P列: KPI判定 ○△×（O列から自動判定）
- Z列: 必須項目チェック

### 自由形式での投稿もOK

フォーマットを使わず自由に書いても、Botがキーワードから自動検出します:

| こう書くと | Botが検出する内容 |
| --- | --- |
| Meta広告でCPA改善のためクリエイティブ変更した | 媒体: Meta広告 / カテゴリ: クリエイティブ変更 / KPI: CPA |
| Google検索広告のターゲティングをリマケ中心に変更 | 媒体: Google広告 / カテゴリ: ターゲティング変更 |
| TikTokの日予算を3万→5万に増額 | 媒体: TikTok広告 / カテゴリ: 予算配分変更 |

※ レコード番号・計測期間が含まれていない場合は、Botがスレッドで確認します。

## `#marketing-knowledge-input` — 外部ナレッジ投入チャンネル

### 概要

自分が打った施策ではないが、チームに共有すべき外部ナレッジを投入するチャンネル。

コンサル資料・媒体公式アップデート・セミナーで得た知見など、「施策記録ではないが、ナレッジとして蓄積すべき情報」を投稿します。

外部ナレッジ投入Bot（2分間隔で巡回）が自動検出し、分類情報を回収してナレッジベースに仮登録します。

### 必須3項目（Botが回収）

| 項目 | シート列 | 選択肢 |
| --- | --- | --- |
| 種類(type) | B列 | casebook / how_to / platform_bp / principle / insight / troubleshoot / regulation |
| 領域(domain) | C列 | クリエイティブ / ターゲティング / 入札 / 予算 / LP / アカウント設計 / レポート / 顧客対応 / 法規制・審査 / 全般 |
| 媒体(medium) | D列 | Meta広告 / Google広告 / YouTube広告 / TikTok広告 / LINE広告 / 横断 |

### 推奨フォーマット

```jsx
種類: platform_bp
領域: クリエイティブ
媒体: Meta広告

ASCキャンペーンでは静止画よりもUGC風動画の方がCPAが30%改善する傾向。
絹川コンサルの分析結果より。90秒以上の長尺が認知目的では有効だが、
獲得目的では15-30秒が最適。
```

フォーマットを使わず自由に書いてもOK。Botがキーワードから自動検出し、不足分はスレッドで確認します。

### Botの動作フロー

1. 投稿を検出 → 分類情報を自動検出
2. 不足があればスレッドで案内（48時間以内に回答）
3. 全項目揃い次第 → ナレッジベースシートに仮登録（review_status: レビュー待ち）
4. `#marketing-knowledge-review` で松岡に通知
5. 松岡が承認 → KB確定 → Notion同期

> **施策記録(PB)を経由せず、ナレッジベース(KB)に直接投入するルートです。**
> 

> **松岡レビュー後に確定するため、投稿時点では仮登録です。**
>

## `#marketing-knowledge-qa` — Q&Aチャンネル

ナレッジベースに蓄積された過去の施策知見に対して、自然言語で質問できます。

AIが関連ナレッジを検索し、回答を生成します。

### 質問パターン

| 質問パターン | 回答内容 |
| --- | --- |
| 〇〇広告のベスプラは？ | その媒体の成功パターン一覧 |
| 前に似たケースあった？ | 類似条件の過去施策と結果 |
| 業種×媒体の勝ちパターンは？ | 業種別の推奨施策 |
| CPAが悪化したときの対処法 | KPI改善に効果があった施策集 |
| #KB-023の詳細を教えて | 特定ナレッジの全情報 |

## `#marketing-knowledge-review` — KB昇格レビュー通知

KB昇格待ちの施策・外部ナレッジのレビュー待ちを松岡に通知するチャンネル。

平日10:00に、以下の2種類のレビュー依頼が届きます:

1. **PB→KB昇格候補**: 担当者確認済み・成果○の施策（打ち手ログS列で承認/却下）
2. **外部ナレッジ投入**: `#marketing-knowledge-input` から仮登録された情報（ナレッジベースO列で承認/却下）

外部ナレッジは投入時にも即時通知されます（レビュー通知Botの定時通知とは別）。
