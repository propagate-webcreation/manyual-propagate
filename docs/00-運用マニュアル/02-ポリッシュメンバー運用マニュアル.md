# 1. ポリッシュとは・担当範囲

ポリッシュは、AI制作案件のサイト制作を行うチームです。**キュー方式**で運用します。特定の案件に専任で張り付くのではなく、タスクキューの上から順に着手し、作業フェーズが完了したら次のタスクに移ります。

## ディレクターが担当する範囲（2点のみ）

- **(a) 初回のデモサイト制作完了時**のチェック・FBのみ → **デモサイトの顧客提出は営業が行う**（営業がデモ実施・提出）
- **(b) 初回のスマホ最適完了時**のチェック・FB → **顧客への提出もディレクターが行う**

## ポリッシュが担当する範囲

- **(a) 上記以外の全工程**
- **(b) 顧客からの修正FBへの対応 → 修正完了後の顧客への再提出**

> 💡 初回のデモサイト提出は営業が行い、スマホ最適提出はディレクターが行います。**顧客からの修正FB後の対応・再提出はポリッシュが直接行います。**
>

# 2. フィルタビューの使い方（必須）

> ⚠️ **制作進行シートでの作業は、必ずフィルタビューを使って行ってください。** 通常の「フィルタ」（漏斗アイコン）は絶対に使わないでください。他のメンバーの画面にも影響してしまいます。
> 

制作進行シートには、ポリッシュ専用のフィルタビューが**既に用意されています。** 自分で新しく作成する必要はありません。

## ポリッシュ用キュー（締切順）

以下のリンクをクリックすると、Q列（ボール保持者）が「ポリッシュ」の案件だけが、フェーズ期限日の近い順に表示されます。

## ポリッシュ用/デモサイト制作キュー（締切順）

デモサイト制作中の案件のみに絞ったキューです。Q列=「ポリッシュ」かつM列=「デモサイト制作中」の案件だけが表示されます。

👉 [ポリッシュ用/デモサイト制作キュー（締切順）を開く](https://docs.google.com/spreadsheets/d/{PRODUCTION_SHEET_ID}/edit#gid=0&fvid=2024818330)

> ⚠️ **新しいフィルタビューを作成しないでください。** 既存のフィルタビューが用意されているので、それを使ってください。フィルタビューの乱立はシートの管理に支障をきたします。

# 3. キュー方式の運用ルール

## タスクの取り方

1. **制作進行シート**を開き、上記の**「ポリッシュ用キュー（締切順）」フィルタビュー**に切り替える
2. **T列（phase_deadline / フェーズ期限日）の昇順**で並んでいるため、**上から順に**期限が近い案件を確認する
3. 着手する案件を決めたら、**R列（active_polish）に自分の名前を入力**する
4. S列（polish_start_time）にGASが自動でタイムスタンプを記録する

> 💡 制作進行シートには、PSからの連携もアノテーションツールからの修正依頼も全て反映されます。制作進行シートを見れば、対応すべきタスクが全て分かります。
> 

> ⚠️ **AI制作のSlack通知は廃止されました（2026-03-13～）。** アノテーター（修正ツール）から修正依頼が入った際のSlack通知機能は廃止されています。AI制作では**制作進行シートのキューのみ**でタスクを把握してください。AI制作の制作チャンネルも今後段階的に廃止されます。
> 

## 重要ルール

- **フェーズ期限日が近い順**に着手する。特別な理由なく案件を飛ばさない
- R列に名前を入れた案件は、そのフェーズの作業が完了するまで責任を持つ
- 作業が完了したら次のキューの案件に移る（同じ案件の次フェーズを待つ必要はない）
- 次のフェーズでは別のポリッシュメンバーが着手して構わない

## 完了時の2つのアクション（必ずセットで行う）

| # | シート | やること |
| --- | --- | --- |
| 1 | **制作進行シート** | M列を次のフェーズのステータスに変更する |
| 2 | **制作進行シート** | **R列（active_polish）の自分の名前を消す**（次のメンバーが着手できるようにするため） |

> ⚠️ **この2つは必ずセットです。** ステータス変更・名前クリアをセットで行うことで、次の担当者がスムーズに着手できます。
> 

## AJ列（test_url）の記入タイミング

> 🚨 **AJ列（test_url）は、M列を「デモサイト制作完了」に変更した後に記入してください。** VercelのテストURLは、制作が完了してから発行されます。「デモサイト制作中」の段階ではURLは存在しません。
> 

| M列ステータス | AJ列（test_url） | 状態 |
| --- | --- | --- |
| デモサイト制作中 | **空欄であること** | まだURLは発行されていない |
| デモサイト制作完了 | **ここでVercel URLを記入** | 制作完了後にURLが発行される |
| デモサイト提出済み以降 | 記入済み | ディレクター・顧客がこのURLを確認する |

> ⚠️ **「デモサイト制作中」の段階でAI列にURLが入っている場合は、M列のステータス更新が漏れている可能性があります。** M列を「デモサイト制作完了」に変更してください。
> 

## AI列（git_repo_url）の使い方と作業開始手順

> 📍 **AI列には、その案件のGitリポジトリURLが記載されています。** 修正・スマホ最適対応時は、必ず以下の4ステップで作業を開始してください。
> 

**✅ 作業開始前の4ステップ（必須）:**

| # | 操作 | 確認内容 |
| --- | --- | --- |
| 1 | **AI列のURL**でGitHubリポジトリを開く | — |
| 2 | GitHubの**description欄**を確認 | 「レコード番号: {A列} / {C列パートナー名} / {LP or CP等}」が記載されている |
| 3 | 制作進行シートの**A列（レコード番号）と一致するか確認** | レコード番号が一致 → 正しいリポジトリ。不一致 → **絶対に作業しない。**ディレクターに確認 |
| 4 | クローンして**作業開始** | — |

> 🚨 **同一パートナーでもLP・CP等の複数リポジトリが存在します。** パートナー名だけで判断せず、必ずレコード番号で照合してください。過去に間違ったリポジトリを修正してしまい、数時間の手戻りが発生した事故があります。
> 

> 🔧 **GitHub Description記入ルール（2026-03-24〜・必須）**
> 

> サイト修正時に、GitHubリポジトリの**Description欄**に以下のフォーマットで記入されているか**必ず確認**してください。
> 

> **フォーマット**: `レコード番号(半角スペース)パートナー名`
> 

> **例**: `16023 株式会社小島建設④`
> 

> 制作進行シートの「レコード番号」と「パートナー名」を同時に選択してDescriptionにコピペしてください。先方名のみ記入されている場合も上記フォーマットに更新してください。
> 

**🔧 AI列が空欄の場合のトラブルシューティング:**

1. **AJ列（test_url）を確認** → Vercel URLの `https://` 以降のサブドメイン部分がリポジトリ名です（例: `https://acme-corp.vercel.app` → リポジトリ名は `acme-corp`）
2. **GitHubのdescription欄**でパートナー名を検索（2月後半以降の案件は記載あり）
3. 上記で特定できない場合 → **R列を「確認待ち」にしてディレクターに確認**
4. 確認後、**AI列にURLを記入**して次回以降のメンバーが迷わないようにする

## Default Setting 最新化手順（作業開始前・必須）

> 🚨 **クローン完了後、修正作業に着手する前に必ず実施してください。** 最新ルールの適用および Tracker.js 保護ルールの反映を目的としています。
> 

**実施タイミング:**

上記4ステップでクローン（またはローカル環境の準備）が完了し、対象案件のプロジェクトを開いた直後。修正作業に着手する前。

**手順（対象案件のリポジトリルートで実行）:**

```
rm -rf .cursor
git fetch https://github.com/propagate-webcreation/DefaultSetting.git
git checkout FETCH_HEAD -- .cursor/
git checkout FETCH_HEAD -- package.json
```

その後、必要に応じて `npm install` ・起動を行い、修正作業を開始してください。

改行調整が必要な場合は、Composer で `/auto-phrase` を実行してください。

> ⚠️ **注意事項**
> 

> - **必ず対象案件のリポジトリルートで実行すること**（`app` や `src` などのサブフォルダでは実行しない）
> 

> - `rm -rf .cursor` は実行中ディレクトリ直下の `.cursor` を削除するため、**実行場所を誤らないこと**
> 

> - `git checkout FETCH_HEAD -- package.json` により `package.json` が上書きされるため、未コミット変更がある場合は事前にバックアップまたは `git stash` を検討すること
> 

> - **作業途中ではなく、作業開始前に実施すること**
>

## 想定以上に時間がかかる案件の対応

以下のようなケースでは、ヒアリングフォーム以外の経路（メール・Googleドキュメント等）で顧客から大量の情報が提供されており、想定以上に時間がかかることがあります。

- 顧客からメール等で明確なサイト構成指示・大量の画像が来ている
- ヒアリングシートの自由記入欄にGoogleドキュメントURL等で詳細要望が入っている

**対応ルール:**

1. **その案件で止まらず、後回しにして次の案件に進行する**
2. **#制作-ai_質問の場** のデモ制作共有スレッドに以下3点を返信でシェアする：
    - レコード番号（A列）
    - パートナー名（C列）
    - 詰まりポイント（何が原因で時間がかかるか）
3. ディレクターからの解決策提示を待ち、回答後に着手する

> ⚠️ **1つの案件に長時間ハマって他の案件の進行が止まることが最も避けたい事態です。** 想定以上に時間がかかると判断したら、すぐに上記の共有フローに切り替えてください。
> 

## 「確認待ち」の使い方（ディレクターにボールを渡す）

作業中に以下のようなケースで、ディレクターに確認や対応を依頼する必要がある場合は、**R列で「確認待ち」を選択**してください。

**対象ケース（例）:**

- 先方からの修正指示の意図がわからず、ディレクターに確認が必要
- 先方との連絡手段がわからない
- 先方への報告・連絡をディレクターにお願いしたい場合
- AI列（GitリポジトリURL）が空欄でリポジトリが特定できない
- 技術的に判断できない事項がある
- 何か事情があって案件を離れなければいけない場合
- 先方への連絡・報告をディレクターにお願いしたい場合

**操作手順:**

| # | 操作 | 説明 |
| --- | --- | --- |
| 1 | **R列**で「**確認待ち**」を選択 | Q列（ボール保持者）が自動的に「**ディレクター**」に切り替わる |
| 2 | **Slackで担当ディレクター（I列）に連絡** | 何を確認したいかを具体的に伝える |
| 3 | **次のキューの案件に進む** | 確認待ちの案件で止まらず、他のタスクを進める |

> 💡 **M列（ステータス）は変更しないでください。** R列を「確認待ち」にするだけで、Q列が自動的にディレクターに切り替わり、ポリッシュのキューから外れます。
> 

> ⚠️ **ディレクターが確認完了後、再度ポリッシュにボールが戻ってきます。** ディレクターがR列にポリッシュメンバーの名前を入れるか、「確認待ち」を解除することでキューに戻ります。その際、元の担当者が続行しても、別のメンバーが着手しても構いません。

# 4. ★ ステータス別アクションガイド

> 🎯 **このセクションの目的**: 制作進行シートのM列（AI統合ステータス）を見て、**「今、自分は何をすべきか」を即座に判断できる**ようにするためのガイドです。
> 

> ⚠️ **提出前チェック 4原則（全フェーズ共通）**
> 

> 1. **実環境確認必須** — ローカル（npm run dev）だけで判断しない。必ずVercelプレビューで確認すること
> 

> 2. **修正ツールのステータスは顧客が見ている** — ステータス変更＝顧客への通知という意識を持つこと
> 

> 3. **チェックリスト通過前の提出禁止** — 急いでいても、各フェーズの✅チェックリストを通さずにステータスを変更しないこと
> 

> 4. **🚨 対応注意案件（AG列 赤文字）はポリッシュから直接提出しない** — AG列（caution_note）に記載がある案件、またはC列（案件名）が赤文字の案件は、修正再提出フェーズであっても**必ずディレクターの確認を通してから提出すること**。R列を「確認待ち」にしてディレクターに連絡してください
> 

> 📖 **FB内容の用語は統一されています。** セクション名・UIパーツ名等、FBで使われる用語の正式名称は以下の用語集を参照してください。

## ① ヒアリング完了

> 📍 **いつこのステータスになるか**: 営業がヒアリングを完了し、デモサイト制作に着手するタイミングです。
> 

**やること:**

1. キューから着手 → **R列**に自分の名前を入力
2. **AH列**（ヒアリングシートURL）を開き、顧客の要望・業種・参考サイト等を確認
3. **M列**を「**デモサイト制作中**」に変更

**✅ 着手前チェック（制作開始前に必ず確認）**

- [ ]  AH列ヒアリングシートの要件を**全て把握**したか（業種・参考サイト・顧客の要望）
- [ ]  不明点・矛盾点はないか（あれば制作開始前にディレクターに確認）
1. 上記チェック通過後、デモサイト制作を開始

> ⚠️ ヒアリングシートの内容に不明点がある場合は、制作を始める前にディレクターにエスカレーションしてください。
> 

> ➡️ **次の工程**: M列 =「デモサイト制作中」（ボール: **ポリッシュ** ＝ 自分で継続）
>

## ② デモサイト制作中

> 📍 **いつこのステータスになるか**: ①でM列を変更した直後。デモサイト制作を進行中の状態です。
> 

**やること:**

1. デモサイト制作を完了させる

**✅ 提出前チェック（ディレクターに渡す前に必ず確認）**

- [ ]  ヒアリングシートの要件を**全て反映**したか（1つずつ照合）
- [ ]  **Vercelプレビュー**で実環境表示を目視確認したか（ローカル確認のみで完了にしない）
- [ ]  お問い合わせフォームが**正常に動作**するか
- [ ]  全リンクが**正常に動作**するか（リンク切れなし）
- [ ]  テキストの**誤字脱字**がないか
- [ ]  画像の表示・解像度に問題がないか

**完了時の4つの操作（必ずセットで実施）:**

| # | 列 | 操作 |
| --- | --- | --- |
| 1 | **M列** | 「**デモサイト制作完了**」に変更 |
| 2 | **AJ列** | VercelのテストURLを記入（※M列変更後にURLが発行される） |
| 3 | **AK列** | **修正ツールURLを記入**（↓ 下記の重要注意を必ず確認） |
| 4 | **R列** | 自分の名前を**クリア** |

> 🚨🚨🚨 **AK列（revision_tool_url）についての重要注意** 🚨🚨🚨
> 

> AK列に入力する修正ツールURLは、**「先方（お客様）に共有するためのリンク」**です。
>

### ④-2. 重度修正の場合（タイムボックス制）

修正内容が大規模な場合は、**90分タイムボックス制**で対応します。90分で対応できる範囲を作業し、残りは+5営業日の追加期間で対応する**分割納品方式**です。

#### 重度修正の定義（以下のいずれかに該当する場合）

| 判定 | 条件 | 例 |
| --- | --- | --- |
| A | **構成変更** | セクションの順序変更、追加・削除の要望 |
| B | **ページ増減** | ページの追加・削除の要望 |
| C | **方向性否定** | 「雰囲気が違う」「イメージと違う」等、デザイン方向性へのFB |
| D | **画像作成提案** | 顧客が画像を用意せず「いい感じに作ってほしい」という依頼 |
| E | **長文詳細指示** | ドキュメントレベルの長文修正指示 |
| F | **体感判断** | 上記に該当しないが、通常の2倍以上の作業量が見込まれる場合 |

#### 対応フロー

1. **スコープ判定**: 修正ツール・メール・LINEを確認し、上記A〜Fに該当するか判定
    
    → 該当しない場合は、通常フローで対応
    
    → 該当する場合は、以下のタイムボックスフローへ
    
2. **90分で対応できる範囲を優先度順に着手**
    
    優先順: **トップページ ＞ 下層ページ**、**構造 ＞ 装飾**
    
3. **90分経過 or 対応可能範囲が完了**したら作業終了
4. **対応済み / 未対応の内容を整理**する
5. **顧客連絡**: 第5章「重度修正時の分割納品テンプレート」を使い、**担当ディレクター名義（I列）で文章を作成・送信**
    
    ※ポリッシュ個人名ではなく、必ずI列の担当ディレクター名で送ること
    
6. **シート更新（通常の3アクション + フェーズ期限再設定）**

| # | 列 | 操作 |
| --- | --- | --- |
| 1 | **M列** | 「**デモサイト修正済み（顧客確認中）**」に変更 |
| 2 | **AJ列** | VercelのテストURLを記入 |
| 3 | **R列** | 自分の名前を**クリア** |
| 4 | **T列** | **フェーズ期限日を現在日から+5営業日に再設定** |

> ⚠️ **重度修正時は、T列（phase_deadline）の再設定を忘れないこと。** 残りの未対応分に対応するための+5営業日を確保します。
> 

> 💡 **ディレクターへの報告は不要です。** ポリッシュが判定→対応→顧客連絡→シート更新まで一貫して行います。

ポリッシュメンバーには、顧客へ直接連絡するための権限が付与されています。

> ・**自分が管理する案件以外は、メール・LINEともに既読をつけないこと**
> 

> ・万が一既読をつけてしまった場合は、メール・LINEともに**「要対応」タブに入れること**
> 

> ⚠️ **再提出時は必ず変更箇所を箇条書きで記載してください。** リンクだけでは顧客がどこを変更したか分からず、見落としが発生します。顧客はアノテーションツールや修正ツールを使って指示していることが多いため、対応した修正内容を明確に伝えることが重要です。
> 

## メール・LINE 書式レギュレーション（必須）

**改行ルール（5つ）:**

| --- | --- | --- |

| ③ | 自己紹介の後 | **空行を入れる** |
| ④ | 本文の段落間 | **空行を入れる** |
| ⑤ | ▼セクション見出しの前 | **空行を入れる** |

# 6. エスカレーションルール

判断に迷うこと、対応に不安があることは**全てディレクターにエスカレーション**してください。

> 🚨 **エスカレーション時は、以下の2つを必ずセットで行ってください。**
> 

> 1. **Slackで担当ディレクター（I列）に直接メッセージ**を送る
> 

> 2. **ディレクター宛て**に業務依頼シートにエスカレーション内容を記入する
> 

> どちらか片方だけでは連携漏れが発生します。**必ず両方**行ってください。
> 

| 状況 | 対応 |
| --- | --- |
| デザインの方向性に不安がある | Slack連絡 ＋ ディレクター宛て業務依頼 → ディレクターに相談 |
| 顧客の修正指示が不明確・理解しがたい | Slack連絡 ＋ ディレクター宛て業務依頼 → ディレクターに相談 |
| 技術的に対応が難しい | Slack連絡 ＋ ディレクター宛て業務依頼 → ディレクターに相談 |
| フェーズ期限日に間に合わない見込み | Slack連絡 ＋ ディレクター宛て業務依頼 → ディレクターに報告 |

> ⚠️ **営業への直接エスカレーションは不要です。** ポリッシュからのエスカレーションは全てディレクターに一本化します。ディレクターが必要と判断すれば、ディレクターから営業に連携します。
> 

## AI自動修正のバグ・質問・フィードバック

AI自動修正（Bot駆動フロー）に関するバグ報告、仕様質問、改善要望は **Slack チャンネル `#制作-ai_自動修正fb`** に投稿してください。

- 対象: 新旧比較画面の不具合、Botが誤修正した事例、画像差し替え対応の相談 等
- 通常のディレクターFB・業務依頼とは**別ルート**で、開発側にダイレクトにフィードバックされます
- 投稿時はレコード番号（A列）・パートナー名（C列）を添えると調査がスムーズです
