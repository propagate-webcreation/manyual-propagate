# アノテーター修正フロー（AI自動修正）

サイトアノテーター（修正ツール）に投稿された修正指示を、AIが自動で読み取り、サイトを修正・デプロイする仕組みです。ポリッシュは修正結果を確認し、AIが対応できなかった部分を手動で補完します。

**管理画面**: [https://www.propagate-annotator.com/admin](https://www.propagate-annotator.com/admin)

---

# 対応フロー

1. 顧客がサイトアノテーター（修正ツール）に修正指示を投稿する
2. M列が「デモサイト自動修正中」に自動遷移する
3. AIがアノテーター上の修正指示を読み取り、サイトを自動修正してデプロイする
4. M列が「デモサイト修正中」に自動遷移し、ポリッシュのキューに表示される
5. ポリッシュが修正結果を確認する
    - アノテーター上で「未対応」「対応中」になっている修正指示について、AIが正しく修正できているかを確認する
    - 問題なければ、該当の修正指示を「修正完了」に変更する
    - AIの修正に問題がある場合は、手動で修正し手動で「修正完了」に変更する

> ⚠️ **M列変更のタイミング・操作者**: 顧客が修正ツールに修正指示を投稿した時点で、M列は「デモサイト自動修正中」に自動遷移します。ポリッシュ/ディレクターが手動で変更する必要はありません。
> 

---

# 対応範囲と制約

- **画像の変更・差し替え**: AIによる自動修正は未対応。画像に関する修正指示はポリッシュが手動で対応する
- **メール・LINEの確認、完了の送信**: ポリッシュが対応する
- **スマホ最適フェーズ**: Bot自動修正の対象外（実機確認が必要なため）。従来通り手動対応を継続する

# 修正結果の確認方法

自動修正の結果は、**ローカルの開発環境（AI修正後の最新コード）** と **アノテーター/Vercelのライブサイト（修正前の状態）** を比較して確認します。

## なぜこの運用になるのか

AI自動修正が入ると、以下の状態になります：

- **Gitリポジトリ**: 最新コード（AI修正後）がpush済み
- **Vercel（ライブサイト）**: まだ1つ前のデプロイ（修正前）が表示されている
- **アノテーター**: Vercelと同じく、修正前の状態が表示されている

つまり、修正前後を比較するには：

| 状態 | 参照先 |
| --- | --- |
| **修正後（最新）** | ローカル開発環境（Gitから最新をpull→起動） |
| **修正前（1つ前）** | アノテーター or Vercelライブサイト |

## 確認手順

1. 対象案件のGitリポジトリをローカルに用意する
    - リポジトリURLは制作進行シートの**AH列（git_repo_url）** に記載されています
    - 初回は `git clone`、2回目以降は `git pull` で最新版にする
2. ローカルで開発環境を起動する（起動コマンドは案件により異なるため、各リポジトリのREADMEを参照）
3. ブラウザを2画面に分けて比較：
    - 左：**アノテーター**（修正前の状態）
    - 右：**ローカル開発環境**（AI修正後の最新状態）
4. 各修正指示ごとに、変更が正しく反映されているかを確認する
5. 問題なければ → アノテーター上で該当の修正指示を**「修正完了」** に変更する
6. 問題があれば → ローカルで手動修正 → push → デプロイ確認 → 手動で **「修正完了」** に変更する

> 💡 **アノテーターは修正前のまま表示されています。** 修正指示の番号マーカー（①②③...）とローカル開発環境を見比べながら、「本当に直ったか」「意図通りに反映されたか」を確認するイメージです。
>
